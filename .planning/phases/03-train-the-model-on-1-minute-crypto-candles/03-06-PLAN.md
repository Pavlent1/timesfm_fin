<objective>
Rerun the Phase 3 BTC one-month fine-tune with dense sampling and a short forecast horizon that matches the actual comparison goal.

Target configuration:
- training symbol set: `BTCUSDT` only
- training slice: one month
- holdout slice: the immediately following month
- prepared-bundle stride: `1`
- training horizon/output: `5`
- post-train comparison backtest: same shape as `run_14.txt` (`horizon_len = 5`, `stride = 5`)

This plan exists because the current successful run proved the infrastructure, but it still used the legacy `128`-step training assumption and the default sparse bundle stride.
</objective>

<context>
Current observed behavior:
- The existing successful run used `horizon_len = 128` and `output_len = 128` because those values are still baked into `configs/fine_tuning.py`, `src/main.py`, and `src/train.py`.
- The prepared one-month BTC bundle produced only `311` windows because `src/training_manifest.py` and the bundle manifest used `stride = 128`.
- The run succeeded only after forcing `batch_size = 1` to fit inside the available RTX 4070 Laptop GPU VRAM.

Expected effect of the requested rerun:
- Moving bundle `stride` from `128` to `1` will increase the one-month BTC training windows from about `311` to about `39,681`.
- Moving the training horizon from `128` to `5` should align the learned objective with the actual 5-step comparison/backtest use case, but it may expose hidden assumptions in the legacy TimesFM/PAX training path.

Primary execution risk:
- The legacy trainer may have real assumptions that `output_len`, `horizon_len`, and/or `output_patch_len` stay at `128` or at least follow a divisibility rule.
- The stride increase will greatly increase training examples and runtime, so the batch and epoch schedule will need to be rebalanced instead of blindly reusing the current settings.
</context>
