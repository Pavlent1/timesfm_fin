<objective>
Rerun the Phase 3 short-horizon fine-tune with three currency pairs instead of one, while preserving the same absolute date ranges and the same BTC evaluation target used by the current one-pair baseline.

Target configuration:
- training symbols: `BTCUSDT`, `ETHUSDT`, `SOLUSDT`
- training slice for each symbol: `2026-02-01T00:00:00Z` to `2026-03-01T00:00:00Z`
- holdout slice for each symbol: `2026-03-01T00:00:00Z` to `2026-04-01T00:00:00Z`
- prepared-bundle stride: `1`
- training context/output: `context_len = 512`, `horizon_len = 5`, `output_len = 5`
- parent checkpoint: same local TimesFM finance checkpoint lineage used by `03-06`
- primary comparison target: the existing one-pair run `btc-one-month-h5-s1-1ep-b1`
- primary report shape: BTC-only report matching `outputs/backtests/btc-one-month-h5-s1-1ep-b1-h5.txt`

This plan exists because `03-06` proved the short-horizon BTC-only path, but the user now wants to test whether adding ETH and SOL in the same training month improves BTC results relative to the one-pair fine-tune.
</objective>

<context>
Current baseline:
- The one-pair short-horizon baseline already exists at `outputs/training_runs/runs/btc-one-month-h5-s1-1ep-b1/`.
- Its prepared bundle used `window_length = 517`, `stride = 1`, and produced `39,804` BTC train windows.
- Its BTC comparison report already exists at `outputs/backtests/btc-one-month-h5-s1-1ep-b1-h5.txt`.

What changes in this rerun:
- The time ranges stay fixed, but the train bundle expands from one symbol to three.
- If all three symbols have comparable coverage, the train-window count should rise from roughly `39,804` to roughly `119,412`.
- The model should still be evaluated on BTC with the same report shape so the comparison answers a single question: did multi-symbol training improve BTC performance?

Primary execution risks:
- Strict cleaning may reject one symbol even if the others are clean enough, so the manifest and bundle report must make missing-data asymmetry explicit.
- Using three symbols will materially increase wall-clock training time and optimizer updates if the run keeps `batch_size = 1` and `num_epochs = 1`.
- A comparison can become misleading if the run quietly changes more than symbol composition; the plan must lock the fairness contract before execution.
</context>
