# Quick Task 260418-qj7: implement per-step conditional backtest metrics for thresholded direction analysis

**Created:** 2026-04-18
**Status:** Ready for execution

## Objective

Add report-level conditional direction-accuracy metrics to the PostgreSQL-backed crypto minute backtest so operators can test whether TimesFM has a higher chance of guessing the final direction correctly when the realized or predicted move from candle 0 to each future candle is large enough.

This task is specifically about the `src/crypto_minute_backtest.py` report path. The goal is to extend the current per-step reporting so it explains direction accuracy for steps 1 through 5 under step-specific move thresholds derived from the existing market-only follow-up analysis.

## Locked Requirements

- Analyze every future candle from step 1 through step 5, not only step 5.
- Keep the current direction-success contract: success is the existing `direction_guess_correct` logic relative to `last_input_close`.
- Evaluate two cohort types for every step:
  - actual move threshold: qualify windows where the realized move from `last_input_close` to `actual_close` is at or above the threshold
  - predicted move threshold: qualify windows where the model-predicted move from `last_input_close` to `predicted_close` is at or above the threshold
- Use two thresholds per step:
  - lower threshold = baseline follow-up deviation minus `0.02` percentage points
  - upper threshold = baseline follow-up deviation plus `0.02` percentage points
- Use the follow-up report at `outputs/backtests/followup_stats_btcusdt_2026-03-16_31d_next5.txt` as the initial source of the baseline deviation values.
- Preserve the existing per-step stats section in the report and append a new conditional-analysis section after it.
- Keep the first implementation in Python report aggregation. Do not add a PostgreSQL schema migration or SQL view change unless a concrete blocker makes the Python path impossible.

## Required Output Shape

Add a new report section after the existing `market_data.backtest_step_stats_vw` table.

Recommended section title:

- `Conditional direction accuracy by move threshold`

Represent the new output as one flat table so it stays easy to diff and test.

Required columns:

- `step_ahead`
- `threshold_basis`
  - expected values: `actual_move_pct`, `predicted_move_pct`
- `threshold_band`
  - expected values: `lower`, `upper`
- `threshold_pct`
- `total_windows`
- `qualified_windows`
- `qualified_share_pct`
- `qualified_correct_count`
- `qualified_accuracy_pct`
- `non_qualified_accuracy_pct`
- `overall_accuracy_pct`
- `accuracy_lift_vs_overall_pct_points`

The most important outputs for the user are:

- how many windows qualify relative to total windows
- what percent of those qualifying windows guessed direction correctly

The remaining columns are required because they make the result interpretable and prevent misleading conclusions from tiny cohorts.

## Implementation Notes

- Add a reusable helper for absolute move-from-input magnitude in `src/backtest_metrics.py`.
  - It should measure the absolute percent move between `last_input_close` and another close.
  - Do not overload `normalized_deviation_pct`, because that helper measures prediction error against the realized close, not move magnitude from the reference candle.
- Keep the conditional aggregation in Python.
  - The needed inputs already exist in each step row: `last_input_close`, `predicted_close`, `actual_close`, `direction_guess_correct`, and `step_index`.
  - Aggregate from the in-memory `window_rows` or from a flattened derivative built from them before report rendering.
- Do not change `db/init/002_phase2_backtest_schema.sql` in this task unless the executor can justify a real need.
- Do not change `src/crypto_minute_followup_stats.py` in this task.
  - Its current report is only the baseline source for the default threshold table.
